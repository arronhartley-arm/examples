Development Studio provides tools that support Arm Linux application development and debug, including shared libraries. This example, based on the "Gnometris" game, illustrates such support.
Note
For information on locating the example files and extracting them for use either on the command-line or in Arm Development Studio, see the installation instructions.

Purpose and scope

This document describes how to build, run and debug the Gnometris example provided with Development Studio, including shared library debug.
Gnometris is a falling-block game, one of a suite of Gnome games downloadable from Gnome Games, and now called Quadrapassel. This example is based on gnome-games-2.26.2.
This example's files can be found in the ...\gnometris directory within Linux_examples.zip. Import this .zip archive into your workspace to access the gnometris project.
Note
This example is intended to be built with Linaro arm-linux-gnueabihf GCC. If you wish to modify and rebuild the example, you must have Linaro arm-linux-gnueabihf GCC installed. Linaro arm-linux-gnueabihf GCC is not supplied with Development Studio, but can be downloaded from the Linaro web-site (see below).
The example is intended to be run or debugged on an Arm Linux target, such as an FVP model of Versatile Express with Cortex-A9x1 Core Tile (Cortex-A9x1 FVP model), Versatile Express Cortex-A9x4...
Pre-built executables are provided that can be used to run and debug the example. To rebuild this example, you must import into your workspace the "Linux Distribution Example" package that can be downloaded from the Downloads on developer.arm.com. This provides the example Linux distribution project/directory, and headers and libraries that are compatible with the example Linux distribution.

Terminology

This document refers to 'host' and 'target' systems. The 'host' system is the Linux or Windows desktop computer that you use for most of your work. The 'target' system is some Arm-based hardware (or model of such hardware) on which the example Arm Linux distribution is running.

Hardware and software requirements

  • A host workstation (Linux or Windows) is required to build the example, communicate with the target, and run the Debugger.
  • A local display device (for example, an LCD monitor) connected directly to your Arm Linux target, or a remote display such as an Exceed or Xming X server running on the host workstation. This is not needed if running the example on a software model such as the Cortex-A9x1 FVP model.
  • A serial terminal emulator such as the Terminal view in Development Studio, PuTTY, TeraTerm, or Minicom for Linux, connected via a serial cable to your platform. This may be needed when running the example on real target hardware, to monitor the Arm Linux boot process and provide a terminal interface to Arm Linux. This is not needed if using a software model as that provides its own terminal. To open the Terminal view in Development Studio, select Window → Show View → Other... to open the Show View dialog box, then expand the Terminal group and select Terminal, and click OK. To configure the terminal settings, click on Open a Terminal in the Terminal view, then select the required connection type (for example, Serial Terminal), then enter the required settings (for example, 38400 baud, 8 bits, no parity, 1 stop bit) and click OK.
  • A secure copy program such as scp, needed to allow files to be copied from host to target. Windows versions of this Linux command are available, such as pscp as provided with PuTTY. Alternatively, use the Debugger's Debug Configurations dialog or Remote System Explorer to transfer files from host to target.
Gnometris is a graphical game in a windowing environment, so a mouse should preferably be connected to the target in addition to a normal keyboard. If using the Cortex-A9x1 FVP model on the host, the mouse and keyboard of the host can be used to control the Gnometris game.

Features of the Gnometris example

The following features make Gnometris an interesting example for illustrating the various tools included in Development Studio and how these work together to provide an integrated Arm Linux application development environment:
  • Gnometris is structured as an application executable gnometris plus a separate application shared library libgames-support.so which is common to all the gnome-games.
  • All the gnometris application source files are written in C++, and all the libgames-support application shared library source files are written in C. This is useful to illustrate the mixed-language software development and debug features of the tools.
  • It uses a large number of system shared libraries (gtk, glib, pango, cairo, atk, etc), which is typical of a real-world Linux application. This is useful to illustrate the shared library support and debug features of the tools.
  • It uses an interrupt-driven timer, to drive the speed at which shapes fall at different skill levels. This is useful to illustrate the exception handling features of the Debugger.
A small number of simplifying changes have been made by Arm Limited to the Gnometris source code for the purposes of this example. These are all marked with _ARM_ to allow them to be easily found.

Installing Linaro GCC arm-linux-gnueabihf as a toolchain in Development Studio

To be able to modify and rebuild this example, download Linaro arm-linux-gnueabihf GCC 4.8 for Armv7/AArch32, and untar it onto your host machine (making a note of the installation directory). Then add it as a toolchain in Development Studio as follows:
  1. In Development Studio, select Window → Preferences → Arm DS → Toolchains.
  2. In the Toolchains dialog, click Add.
  3. In the Select Toolchain Path dialog, enter the path to the toolchain binaries. This is normally the /bin sub-directory of the installation directory noted above. Then click Next.
  4. In the Discovered Toolchain Information dialog, check the toolchain information is correct, then click Finish.
  5. The new toolchain appears in the Toolchains dialog, click Apply, then click Restart.

Building this example

A pre-built executable is provided for hard-float filesystems. The hard-float executable (gnometris, compiled with -mfloat-abi=hard) is targeted at the hardware floating point ABI, as found in, for example, the Linaro 12.04 and later releases.
This example is intended to be built with a GCC cross-compiler for Arm Linux target, such as the Linaro GCC toolchain, that must be downloaded, installed, and added as a toolchain as described earlier.
This example can be rebuilt using the Eclipse (makefile builder) project supplied, or directly on the command-line using the Makefile supplied with the make utility.
To rebuild this example, you must have imported into your workspace the "Linux Distribution Example" package that can be downloaded from the Downloads on developer.arm.com. This provides the example Linux distribution project/directory, and headers and libraries that are compatible with the example Linux distribution. The distribution project does not need to be built (actually, it cannot be built).
For a good debug view, the compiler's optimization level is set to -O1 in the Makefile. You can change this to raise the optimization level to -O2 or -O3 for higher performance code generation, but at the cost of a worse debug view.
The build produces both stripped and unstripped versions of the ELF executable gnometris and a shared library libgames-support.so. The unstripped (debug) versions contain debug information for loading into the Debugger on the host. Smaller stripped (no debug) versions for downloading to the target are created by stripping off the debug information.
Note that the other build files provided as part of gnome-games-2.26.2, such as Makefile.am, Makefile.ini, xmldocs.make and gnome-doc-utils.make are not used here.

Building on the command-line

To build on the command-line with the supplied make utility:
  • On Windows, open an Arm DS Command Prompt from the Start menu, run the select_toolchain utility, and select GCC 4.8.3 [arm-linux-gnueabihf] from the list
  • On Linux, run the suite_exec utility with the --toolchain option to select the compiler and start a shell configured for the Development Studio environment, for example: ~/developmentstudio/bin/suite_exec --toolchain "GCC 4.8.3 [arm-linux-gnueabihf]" bash
Then navigate to ...\gnometris then type:
make
The usual make rules: clean, all and rebuild are provided in the Makefile.

Building with Arm Development Studio

  1. In the Project Explorer view, select the project you want to build.
  2. Select Project → Build Project.
The gnometris Eclipse (makefile builder) project is used to build the executable gnometris and shared library libgames-support.so.

Running/debugging this example on the Cortex-A9x1 FVP model

To run this example on the Cortex-A9x1 FVP model as supplied with Development Studio:
  1. First download the "Linux Distribution Example" package from Downloads on developer.arm.com then either unzip it into a local folder on your hard-disk or import it into your workspace
  2. On Windows, from the Start menu, open an Arm DS Command Prompt, then navigate to the distribution folder where the Linux Distribution was unzipped
  3. On Linux, navigate to the distribution folder where the Linux Distribution was unzipped
  4. Boot Linux on the Cortex-A9x1 FVP model with:
    FVP_VE_Cortex-A9x1 -a "boot\FVP\FVP_VE_V7\FVP_VE_V7.axf" --data "kernel\linux-linaro-3.4-rc3-2012.04-0-patched\built\VE_V7\arch\arm\boot\Image@0x80008000" --data "boot\FVP\FVP_VE_V7\rtsm_ve-cortex_a9x4.dtb@0x80f00000" -C motherboard.mmc.p_mmc_file="boot\FVP\FVP_VE_V7\linaro-image-alip-genericarmv7a-20150323-328.rootfs.tar.gz_VE_V7.image" -C cluster.cpu0.semihosting-enable=0 -C motherboard.terminal_3.start_telnet=false -C motherboard.terminal_3.mode=raw -C motherboard.pl011_uart3.untimed_fifos=true -C motherboard.terminal_3.start_port=5003 -C motherboard.smsc_91c111.enabled=1 -C motherboard.hostbridge.userNetworking=1 -C motherboard.hostbridge.userNetPorts="5555=5555,22=22,8080=8080"
    Port 5555 is used for debug, 22 for sshd, and 8080 for httpd.
    For Linux hosts, you may need to use "8022=22" for sshd. If so, the Remote Systems configuration must be adjusted accordingly (see below)
  5. Wait for the Linux boot to complete and for the Linux Desktop to appear in the Fast Models CLCD
  6. In Development Studio, open a Remote Systems view by using Window → Show View → Other → Remote Systems → Remote Systems
  7. In the Remote Systems view, click on Define a connection to remote system. Select SSH Only, then click Next. In the Host name field, enter "localhost" then click Finish
  8. If you used "8022=22" for sshd, then you must adjust the Remote Systems configuration accordingly. In the Remote Systems view, right-mouse-click on Sftp Files, then select Properties. Select Subsystem, change the port number from the default 22 to 8022, then click OK
  9. In the Remote Systems view, expand Sftp Files, then expand Root. In Enter Password, enter User ID="root", then click OK. Expand /home/root. The contents of the /home/root folder is displayed, initially empty
  10. Right-mouse-click on Ssh Terminals and select Launch Terminal. The "root@genericarmv7a" prompt appears in the Terminals view
  11. From Run → Debug Configurations..., select gnometris-FVP, then click Debug. This is pre-configured to connect to Linux on the Cortex-A9x1 FVP model, download the stripped application and shared library to /home/root/, start gdbserver on the target, load the debug information from the debug/unstripped version of the image into the Debugger, then start executing the application, stopping at main()
  12. Click Continue or press F8. Gnometris appears in the Fast Models CLCD
  13. Click Disconnect from Target
  14. You can run (not debug) the stripped image again by launching it from the Terminals view with ./gnometris --display=:0.0

Running/debugging this example on a hardware target

To run this example on a hardware target:
  1. If you have not done so already, boot Linux on your hardware target and log-in as root
  2. Ensure it is connected on Ethernet and get its IP address with ifconfig
  3. In Development Studio, open a Remote Systems view by using Window → Show View → Other → Remote Systems → Remote Systems
  4. In the Remote Systems view, click on Define a connection to remote system. Select SSH Only, then click Next. In the Host name field, enter the IP address obtained earlier, then click Finish
  5. In the Remote Systems view, expand Sftp Files, then expand Root. In Enter Password, enter User ID="root", then click OK. Expand /home/root. The contents of the /home/root folder is displayed
  6. Right-mouse-click on Ssh Terminals and select Launch Terminal. The Linux prompt appears in the Terminals view
  7. From Run → Debug Configurations..., select gnometris-gdbserver, then click Debug. This is pre-configured to connect to Linux on your hardware target, download the stripped application and shared library to /home/root/, start gdbserver on the target, load the debug information from the debug/unstripped version of the image into the Debugger, then start executing the application, stopping at main()
  8. Click Continue or press F8. Gnometris appears in the Fast Models CLCD
  9. Click Disconnect from Target
  10. You can run (not debug) the stripped image again by launching it from the Terminals view with ./gnometris. You may need to set the LD_LIBRARY_PATH and/or DISPLAY first - see below

Optional Components

The Gnometris application can make use of additional icons and background graphics. To optionally install these, copy the following files:
  • To have an icon appear in Help → About, copy gnome-gnometris.svg
    from ...\gnometris\gnome-games-2.26.2\icons\scalable\
    to /usr/share/icons/hicolor/scalable/apps/
  • To have a background graphic appear in the playing area, create a new directory /home/root/pkgdata/gnometris/pixmaps/ (the first part of this path is configurable at compile time with -DPKGDATADIR) then copy gnometris.svg
    from ...\gnometris\gnome-games-2.26.2\gnometris\
    to /home/root/pkgdata/gnometris/pixmaps/
The game is still playable even if these files are not copied, but there will be a blank background instead of a graphic.
Configuration information (for example, the game level) is saved on exit automatically by writing to the files ~/.config/gnome-games/gnometris and ~/.config/gnome-games/gnometris.accels.

Running Gnometris manually

Instead of using the supplied gnometris-FVP or gnometris-gdbserver debug configurations, Gnometris can be run directly from the target's command-line.
You may need to set the LD_LIBRARY_PATH and/or DISPLAY first, e.g.:
export LD_LIBRARY_PATH=/path where /path is the location of libgames-support.so. (If gnometris and libgames-support.so are in the same directory, LD_LIBRARY_PATH does not need to be set because they are linked with -rpath='$ORIGIN')
export DISPLAY=:0.0 for a local display or, for example, export DISPLAY=10.1.201.33:0.0 to open its window on the host's X server, where 10.1.201.33 is the host's IP address
./gnometris &
Alternatively, these last two commands can be combined into one ./gnometris --display=10.1.201.33:0.0 &
The Gnometris application will start running, and its graphical display appears on the target display after a few seconds.

Playing Gnometris

To start a game, select Game → New using the mouse or keyboard. The basic game keys are: left arrow to move a block left; right arrow to move a block right; up arrow to rotate a block; pause key to drop a block; down arrow to move a block down; space bar to pause the game. The keys can be changed in the Gnometris application's Settings → Preferences → Controls tab.
When running on the Cortex-A9x1 FVP model, two mouse pointers (one for the host, one for the target) appear on the display. To align them, hide the host mouse pointer by pressing the left Ctrl+Alt keys. Press these keys again to redisplay the host mouse pointer. Only the left Ctrl key is operational, the Ctrl key on the right of the keyboard does not have the same effect. If you prefer to use a different key combination, use the trap_key configuration option. Refer to the model's visualisation parameter documentation for details.

Debugging the Gnometris application

Use the supplied gnometris-FVP or gnometris-gdbserver debug configurations to establish a connection to the Cortex-A9x1 FVP model or a hardware target respectively. These will also start gdbserver on the target, load the debug information from the debug/unstripped version of the image into the Debugger, then start executing Gnometris, stopping at main().
You can then set breakpoints in the application, view variables, view memory, view backtrace, etc, as illustrated in the following example session:
  1. Set a breakpoint in the Gnometris application on the BlockOps::dropBlock() function in blockops-noclutter.cpp, with:
    break BlockOps::dropBlock, or
    Use the Project Explorer view to locate and open the file ...\gnometris\gnome-games-2.26.2\gnometris\blockops-noclutter.cpp, then double-click on the BlockOps::dropBlock() function using the mouse.
  2. View the breakpoints set in the Breakpoints view
  3. Resume Gnometris execution, by clicking on the green Continue arrow in Debug Control, or press F8. The Gnometris application will start running, and its graphical display appears on the target display after a few seconds. With the mouse or keyboard, select Game → New to start a game.
  4. "Drop" the current shape, by pressing the pause key on the target keyboard. Note that "Drop" is different to "Move down", depending on your keyboard settings in the Gnometris application's Settings → Preferences → Controls. Execution will stop in BlockOps::dropBlock().
  5. View the call stack, either in the Stack view, or by typing backtrace in the Commands view.
  6. Use Step Over (F6) to step over to line 222.
    Notice the code view follows the PC, and the Register view changes.
  7. View the local variable count in BlockOps::dropBlock() in the Variables tab, or with:
    print count (or its abbreviation p count).
    Click on Step Over twice, and observe the value of count incrementing, to land back on line 222.
  8. Use Step In (F5) to step into the called function BlockOps::moveBlockDown()
    Notice the code view follows the PC, and the Register view changes, and the call stack changes.
  9. Use Step Out (F7) to step out of BlockOps::moveBlockDown()
    Notice the code view follows the PC, and the Register view changes.
  10. Change the "next block" global variable
    set blocknr_next = 5
    5 gives a straight 4x1 block, 6 gives a square 2x2 block. The next block displayed will be updated the next time the display is refreshed.
See Overview: Breakpoints and Watchpoints in Arm Development Studio User Guide for more information.

Debugging the Gnometris application shared library

Being connected to gdbserver, the Debugger will have automatically switched into --target_os=linux mode, to allow debug of shared libraries.
To debug a shared library at source-level, debug information for the shared library needs to be loaded into the Debugger. Once this is done, you can set breakpoints in the shared library, view variables, view memory, view backtrace, etc, in the same way as for the application previously, as illustrated in the following example session:
  1. Disconnect then reconnect to the target to restart the Gnometris application
  2. To see which shared libraries are currently loaded on the target, and whether debug information has been made available for them on the host, use:
    info shared
    To get an interactive list of all possible options for a command such as info, enter on the command-line info followed by Ctrl+Space keys.
  3. Set a breakpoint on the libgames-support.so shared library initialization code
    break games_runtime_init
    Observe the breakpoint appears in the Breakpoints view.
    Restart program execution. Execution will stop on the breakpoint in games-runtime.c
  4. Set a breakpoint in the score handler in the shared library
    break games_scores_add_score
    Restart program execution. This function is called (and the breakpoint will be hit) in games-scores.c at the end of the game to check if you got a high score (if it was not zero).

Known issues and troubleshooting

  • When attempting to build this example, if Eclipse or make reports: fatal error: glib.h: No such file or directory.
    This will occur if you have not imported the example Linux distribution project/directory, or not unzipped all the examples, or accidentally deleted distribution\filesystem\. The example Linux distribution project/directory is provided in the "Linux Distribution Example" package that can be downloaded from the Downloads on developer.arm.com.
  • When importing the example Linux distribution example on Windows, you will see, for example:
    Problems were encountered during import:
A resource exists with a different case: /distribution/filesystem/linaro_hf/usr/include/linux/netfilter_ipv6/ip6t_HL.h
    This is expected behaviour when unpacking the Linux kernel on Windows. The Linux kernel sources are intended to be unpacked, used and built on a Linux platform, not on Windows. This problem is caused by Windows not being able to differentiate between upper and lower case in filenames (e.g. foo.h versus foo.H), where Linux treats these as separate files.
  • Semantic error (red marks) are reported in the Editor view for gboolean, GError and other source keywords, or reports Unresolved inclusion for headers such as <gtk/gtk.h>.
    This will occur if Eclipse cannot locate the header files for the graphics libraries supplied in the example Linux distribution project/directory. The example Linux distribution project/directory is provided in the "Linux Distribution Example" package that can be downloaded from the Downloads on developer.arm.com. You can resolve this by right-clicking on the project in the Project Explorer view and then selecting Properties... from the context menu. In the dialog box, select: C/C++ General → Paths and Symbols → Add... and then add the path to the graphics headers on your host workstation, for example, ...\distribution\filesystem\linaro_hf\usr\include\gtk-2.0\, ...\distribution\filesystem\linaro_hf\usr\include\glib-2.0\, and others specified in the makefile. Select both Add to all configurations and Add to all languages, then click on Apply so that Eclipse will rebuild the index.
  • The Linux target will report: # Cannot exec ./gnometris: Permission denied.
    if you have not set execute permissions on the application. Use, for example, chmod +x gnometris.
  • The Linux target will report: Connection Failed: Failed to delete file /home/root/(app): Permission denied.
    if you don't have permissions to write in the root folder. Modify your Debug Configuration, and set, in Files Tab, a writeable folder in Target download directory:.
  • Breakpoints are not being hit in an application or shared library:
    Ensure the application and any shared libraries on your target match the ones on the host. The code/data layout must be identical, though the application/shared library on your target does not need to contain debug symbols, i.e. these can be stripped to reduce image size. Try copying the application/shared library across from host to target again.
  • On Windows 7 and later, launching a debug configuration to connect to a Fixed Virtual Platform (FVP) model may give the following error: Windows cannot find "C:\Windows\System32\telnet.exe" or No telnet executable was found on your system. Arm FVP models make use of "telnet" as a serial terminal, to enable serial data to be transferred from application code running on an Arm FVP via a modelled UART to a serial terminal. This error occurs when the telnet client is disabled or otherwise unavailable on your computer. The telnet client is not enabled by default on Windows 7 and later. To enable the telnet client on Windows 10:
    1. Right-click on the Start menu, then select "Settings"
    2. Search for "Telnet", and select "Turn Windows features on or off"
    3. From the list that appears, tick the "Telnet Client" checkbox
    4. Click "OK" to close the dialog.
